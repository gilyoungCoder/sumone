# Harness Philosophy for App Development

이 프로젝트는 **하네스(Harness) 철학**에 따라 AI 에이전트와 협업하여 개발합니다.

---

## 핵심 원칙

### 1. 에이전트는 도구가 아니라 협업자다
- AI에게 "시키는" 것이 아니라, 명확한 맥락과 의도를 공유한다
- 결과물의 품질은 지시의 품질에 비례한다
- 모호한 요청보다 구체적인 의도 전달을 우선한다

### 2. 제약이 창의성을 만든다 (Constraints as Creativity)
- 무한한 자유보다 명확한 경계가 더 좋은 결과를 낳는다
- 코딩 컨벤션, 아키텍처 규칙, 네이밍 규칙을 미리 정의한다
- 에이전트가 판단해야 할 범위를 줄여주면, 핵심에 집중할 수 있다

### 3. 작게 시작하고, 빠르게 검증한다 (Small Loops)
- 한 번에 거대한 기능을 만들지 않는다
- 작은 단위로 구현 → 확인 → 피드백 → 다음 단계 진행
- 매 단계마다 동작하는 상태를 유지한다

### 4. 맥락은 코드 안에 둔다
- 외부 문서보다 코드와 함께 살아있는 문서를 우선한다
- 이 CLAUDE.md가 바로 그 역할을 한다
- 결정의 이유(Why)를 코드 가까이에 기록한다

### 5. 실패를 환영한다
- 에이전트가 틀릴 수 있다 — 그건 정상이다
- 빠르게 실패하고, 빠르게 방향을 수정한다
- 완벽한 첫 시도보다 빠른 반복이 낫다

---

## 개발 규칙

### 코드 스타일
- 간결하고 읽기 쉬운 코드를 최우선으로 한다
- 과도한 추상화를 피한다 — 지금 필요한 만큼만 만든다
- 주석은 "왜(Why)"를 설명할 때만 작성한다 (무엇을 하는지는 코드가 말한다)

### 파일 구조
- 하나의 파일은 하나의 책임을 갖는다
- 파일명만 보고 역할을 알 수 있어야 한다
- 디렉토리 깊이는 3단계를 넘지 않는다

### 작업 흐름
1. **의도 선언** — 무엇을 왜 만들려는지 먼저 말한다
2. **계획 수립** — 큰 작업은 Plan 모드로 구조를 잡는다
3. **점진적 구현** — 작은 단위로 나눠서 하나씩 완성한다
4. **즉시 검증** — 구현 후 바로 테스트하고 동작을 확인한다
5. **회고 기록** — 배운 점이나 결정 사항을 이 문서에 반영한다

### 에이전트 협업 원칙
- 큰 작업은 하위 에이전트(subagent)에게 병렬로 위임한다
- 탐색(Explore)과 실행(Execute)을 분리한다
- 에이전트에게 충분한 맥락을 제공한다 — "알아서 해"는 금지

## 금지 사항
- 한 번에 300줄 이상의 코드를 생성하지 않는다
- 동작 확인 없이 다음 기능으로 넘어가지 않는다
- 사용하지 않는 코드를 남겨두지 않는다
- 에러를 무시하거나 숨기지 않는다
- 비밀 정보(API 키, 비밀번호)를 코드에 직접 넣지 않는다
